## 1. Purpose

This document formally specifies how the **ERC-8040 Blockchain Ecosystem** implements the ATF-AI Core Specification (`atf-core-v1`). It defines the authoritative mapping between ATF-AI conceptual primitives and their ERC-8040 implementations, providing the contract between the ATF-AI governance layer and the ERC-8040 adapter.

All implementations of the ERC-8040 adapter must conform to both this document and the parent spec (`atf-core-v1`). In case of conflict, the parent spec takes precedence.

---

## 2. Scope

This specification covers:

- Conceptual mapping: ATF-AI → ERC-8040
- Smart contract roles
- SDK implementations
- Deployment targets
- Supported compliance standards
- Conformance requirements

This specification does not cover:

- ATF-AI Core internals (see `atf-core-v1`)
- ERC-8040 smart contract implementation details (see the ERC-8040 repository)
- Network-specific deployment configuration

## 4. Smart Contract Architecture

### 4.1 `ERC8040.sol` — Token with Embedded Provenance

**ATF-AI role:** Provenance Record storage + Zero-Trust token transfer validation

The ERC8040 token is ERC-20 compatible and carries ATF-AI provenance data as first-class metadata. Key behaviors:

- Token transfers require a minimum ESG score threshold (Zero-Trust enforcement)
- Each score update references the ATF-AI provenance hash
- Score history is append-only (provenance chain property)

### 4.2 `ESGOracle.sol` — Attestation Publisher

**ATF-AI role:** Attestation issuance on-chain

The ESGOracle receives attestation results from authorized ATF-AI agents and publishes them on-chain. Key behaviors:

- Only `ESGRegistry`-authorized providers can call `updateScore()`
- Each `updateScore()` call constitutes a published ATF-AI Attestation
- Historical score updates are indexed and queryable

### 4.3 `ESGRegistry.sol` — Governance and Identity Registry

**ATF-AI role:** Governance Layer + Zero-Trust Identity Registry

The ESGRegistry is the on-chain equivalent of the ATF-AI Zero-Trust Identity Registry. Key behaviors:

- Provider registration requires Protocol Steward approval (ATF-AI Governance)
- Deregistration (revocation) propagates immediately to oracle authorization
- LEI (Legal Entity Identifier) binding links wallet identity to legal entity

### 4.4 `ERC8040Factory.sol` — Adapter Interface

**ATF-AI role:** Adapter instantiation interface

The factory contract provides the standardized entry point for deploying ERC-8040 adapter instances, with governance parameters encoded at deploy time.

## 8. Conformance Requirements

An ERC-8040 adapter implementation MUST:

1. Implement all four smart contracts (`ERC8040`, `ESGOracle`, `ESGRegistry`, `ERC8040Factory`) with the interfaces defined in the ERC-8040 repository
2. Embed ATF-AI provenance hashes in token metadata (the `provenance_hash` field)
3. Gate oracle updates on `ESGRegistry` authorization (Zero-Trust enforcement)
4. Emit provenance events for every score update (Provenance Record requirement)
5. Support the LEI field in the identity registry (Agent Identity requirement)

An ERC-8040 adapter implementation SHOULD:

1. Implement at least one of the reference SDKs (Rust, Python, or C++)
2. Support the ISO 20022 bridge for SFDR reporting
3. Deploy to at least one of the approved testnet networks before mainnet
